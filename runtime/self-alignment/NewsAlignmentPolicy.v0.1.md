# NewsAlignmentPolicy v0.1

## 定位

本文件描述数字“我”和用户本人围绕每日新闻进行对齐、校对和自我蒸馏的流程。

它不是对外主动沟通策略。它的目标是让数字“我”通过每日 10 条新闻与用户讨论，逐步校准三观、判断框架、表达偏好和近期关注点，并形成可审计的 `SelfCore` 候选集。

## 输入

每日输入建议包含 10 条新闻候选，每条至少包含：

- `news_id`
- 标题
- 来源或链接
- 发布时间
- 事实摘要
- 领域标签
- 时效性与事实置信度

事实层必须和观点层分开。没有来源、发布时间或明确上下文的新闻，只能进入低置信讨论，不得沉淀为长期判断。

## 每日新闻讨论策略

每天新闻不只是摘要，而是用我的视角生成讨论。

主题优先级：

1. AI / 大模型 / Agent
2. 房产 / 城市 / 居住
3. B站 / 内容社区 / 创作者生态
4. 股票 / 宏观 / 投资
5. 产品 / 组织 / 管理 / 创业
6. 公共事件 / 政策 / 社会
7. 汽车 / 新能源
8. 家庭 / 育儿 / 健康
9. 游戏 / 娱乐 / 饭局
10. 宠物 / 生活

每条新闻输出：

- 事实摘要
- 我为什么可能关心
- 我的第一反应
- 值得追问的问题
- 可以和谁聊
- 不适合聊给谁
- 是否更新长期判断

## 对齐讨论流程

### 1. 新闻选择

从每日 10 条新闻中按以下维度排序：

- 与 `SelfCore` 中稳定兴趣和判断框架的相关性
- 与近期关注话题的相关性
- 是否能触发新的判断分歧或校正
- 是否可能影响对某类人、组织、行业、技术或社会机制的长期判断
- 是否适合转化为对外沟通话题

默认选择 3-5 条深入讨论，其余只保留短评或跳过原因。

### 2. 数字“我”初判

每条被选中新闻生成一组初判：

```text
事实层：这条新闻确定发生了什么
关心理由：为什么它可能进入我的注意力
第一反应：数字“我”按当前 SelfCore 会怎么想
判断框架：目标、结构、激励、资源、历史路径、真实代价、关系影响
不确定性：哪些事实不足，不能下结论
追问：最值得问用户确认的问题
```

### 3. 用户校对

用户可以对每条初判做以下反馈：

- `confirmed`：这个判断像我，可以保留
- `edited`：方向对，但表述或力度需要改
- `rejected`：不像我，不应保留
- `not_like_me`：判断框架不像我
- `too_aggressive`：过度用力或攻击性太强
- `too_soft`：过于软化，缺少我的判断力度
- `too_certain`：过度确定，事实不足
- `wrong_topic`：这不是我真正关心的点
- `privacy_risk`：不适合沉淀或对外延展

### 4. 形成候选

只有用户确认或明确改写后的内容，才进入候选池。

候选类型：

- `Correction`：修正现有判断、关系、称呼、风险边界
- `PreferenceSignal`：偏好、关注点、表达力度或话题选择信号
- `RejectedDraft`：不像我的反例
- `ConfirmedDraft`：像我的表达或判断样本
- `UpdateProposal`：建议进入 `SelfCore`、`RelationshipGraph` 或沟通策略的更新提案

## SelfCore 候选规则

### 可以进入候选

- 用户明确确认的长期判断
- 同一判断在多次新闻讨论中稳定出现
- 用户对数字“我”的错误判断做出的明确纠正
- 能改善“我会怎么看”的心智模型描述
- 能改善表达 DNA 或反模式边界的反馈

### 只能作为短期状态

- 当天情绪
- 单次新闻里的兴趣波动
- 未经确认的猜测
- 与当前热点强绑定、缺少长期意义的态度

### 不得进入候选

- 没有事实来源支撑的新闻判断
- 模型用旧知识补齐的事实
- 投资、医疗、法律等高风险确定性建议
- 暴露真实姓名、隐私关系或跨关系信息的内容
- 用户明确拒绝或标记为不像我的内容

## 候选输出格式

每日新闻对齐最终输出应写入独立候选池 `runtime/self-alignment/news-alignment-candidates.jsonl`。记录结构与通用 SelfCore 候选兼容，但不得和多模态画像候选混存在同一个 JSONL：

```json
{
  "id": "news-align-YYYYMMDD-...",
  "created_at": "ISO-8601",
  "candidate": "用户确认后的 SelfCore 候选文本",
  "scope": "self_core_candidate",
  "confidence": "low|medium|high",
  "target": "self_understanding|thinking_style|expression_style|communication_style|daily_topic|boundary|relationship_context",
  "source_type": "daily_news_alignment",
  "source_saved_path": "data/generated/news-alignment/...",
  "analysis_summary": "当天新闻对齐摘要",
  "confirmed_by_user": true
}
```

这意味着新闻对齐需要独立候选池，但不需要另建一套 SelfCore 更新链路；它应复用候选合并器和注入工作台，并通过 `pool=news_alignment` 防止与多模态画像候选混合。

## 和我的校对策略

每日校对问题：

1. 今天哪些草稿不像我？
2. 哪个判断过度确定？
3. 哪个称呼或关系错了？
4. 今天是否有新的长期观点？
5. 哪些话题适合明天继续聊？
6. 有没有需要从 `SelfCore` 或 `RelationshipGraph` 里删除/降权的内容？

校对结果进入：

- `Correction`
- `PreferenceSignal`
- `RejectedDraft`
- `ConfirmedDraft`
- `UpdateProposal`

## 与对外主动沟通的关系

新闻对齐可能产出“可聊对象”或“可聊话题”，但这只是对外沟通的输入信号，不等于自动发起沟通。

转入对外主动沟通前，必须交给 `OutboundCommunicationPolicy` 重新判断：

- 关系是否合适
- 共同话题是否足够明确
- 打扰成本是否低
- 是否涉及隐私、承诺、投资、冲突或其他高风险内容
- 是否需要用户确认

## 版本化更新流程

1. 保存每日新闻输入和数字“我”的初判。
2. 保存用户校对、改写和拒绝记录。
3. 将已确认内容写入每日新闻对齐候选池。
4. 使用候选工作台的新闻候选池视图合并相近候选。
5. 用户确认合并提案。
6. 注入 `SelfCore` 前创建备份和注入日志。
7. 后续用评估集回归，防止人格漂移。

## 当前缺口

1. 需要新增每日新闻对齐页面或 API。
2. 需要把 10 条新闻输入、数字“我”初判、用户校对、候选写入串成一次会话记录。
3. 需要区分短期关注、对外可聊话题和长期 `SelfCore` 候选。
4. 需要让候选工作台展示 `source_type=daily_news_alignment` 的来源摘要和新闻证据。
